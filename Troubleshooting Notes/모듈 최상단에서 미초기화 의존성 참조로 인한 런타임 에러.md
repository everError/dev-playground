**Title:** 모듈 최상단에서 미초기화 의존성 참조로 인한 런타임 에러

**Category:** Frontend

**Stack:** Vue, TypeScript

**Status:** Resolved

**Priority:** Medium

---

## Environment

- Framework: Vue 3
- 모듈 번들러: Vite
- 모듈 최상단에서 `export const`로 외부 의존성을 참조하는 코드가 포함된 구조

## Symptom

앱 실행 시 특정 모듈에서 `Cannot read properties of undefined` 또는 `xxx is not initialized` 류의 런타임 에러가 발생한다. 코드 자체에는 문법 오류가 없고, 해당 모듈을 단독으로 보면 문제가 없어 보이지만 앱이 시작되는 순간 에러가 터진다.

## Cause

모듈 로딩 시점과 애플리케이션 초기화 시점의 차이를 이해하지 못한 것이 원인이다.

`import`문으로 모듈을 가져오는 순간 해당 모듈의 최상위 코드가 즉시 실행된다. `export const`로 선언된 변수도 이 시점에 평가된다. 그런데 대부분의 프레임워크는 플러그인 설정, 환경 변수 주입 등의 초기화 작업을 `main.js`(또는 `main.ts`)에서 앱이 시작될 때 수행한다.

문제는 모듈 로딩이 프레임워크 초기화보다 먼저 일어난다는 것이다. 모듈 최상단에서 아직 초기화되지 않은 의존성을 참조하면, 해당 객체가 `undefined`인 상태에서 프로퍼티에 접근하게 되어 에러가 발생한다.

예를 들어 `vue-i18n`의 경우, `main.js`에서 i18n 인스턴스를 생성하기 전에 다른 모듈에서 번역 함수 `t`를 최상단에서 꺼내 쓰려고 하면 실패한다.

```javascript
// 이 파일이 import되는 순간 실행됨
// 이 시점에 i18n은 아직 초기화 전 → 에러
import { i18n } from "./main";
export const t = i18n.global.t;
```

## Solution

외부 의존성에 접근하는 코드는 모듈 최상단이 아니라, 해당 의존성이 준비된 이후 시점에 실행되도록 작성해야 한다.

**원칙 1. 프레임워크가 제공하는 공식 훅을 사용한다**

```javascript
// 컴포넌트 라이프사이클 안에서 안전하게 접근
import { useI18n } from "vue-i18n";

const { t } = useI18n(); // setup() 내에서 실행됨
```

프레임워크가 제공하는 composable이나 훅은 내부적으로 초기화 시점을 보장하므로 안전하다.

**원칙 2. 모듈 최상단에 값을 고정하지 말고 함수로 감싼다**

```javascript
// Bad: import 시점에 즉시 평가
export const api = new ApiClient({ apiKey: process.env.API_KEY });

// Good: 호출 시점에 평가
export const createApi = () => {
  return new ApiClient({ apiKey: process.env.API_KEY });
};
```

함수로 감싸면 실제 호출 시점까지 평가가 지연되어, 초기화가 완료된 이후에 안전하게 의존성을 참조할 수 있다.

**원칙 3. 모듈 최상단은 import에만 집중한다**

모듈 상단에서는 의존성을 가져오는 역할만 하고, 실제 기능 실행은 해당 기능이 필요한 컴포넌트나 함수 내부에서 처리한다.

## Notes

- 이 문제는 Vue에 국한되지 않고, React, Angular 등 모든 모듈 기반 프레임워크에서 동일하게 발생할 수 있다
- 번들러(Vite, Webpack)의 트리 셰이킹과도 관련이 있다. 사이드 이펙트가 있는 모듈 최상단 코드는 트리 셰이킹에서도 불리하다
- 코드 한두 줄 줄이려고 모듈 최상단에 값을 고정하는 것보다, 실행 시점을 명확히 제어하는 것이 유지보수와 안정성 측면에서 훨씬 유리하다
